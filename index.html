NestedScrollView 嵌套ViewPager 在嵌套 RecyclerView

根据业务的需要, 设想了一种NestedScrollView中 嵌套ViewPager, 在ViewPager中继续嵌套RecyclerView，然而当RecyclerView包含多个ItemType时,左右滑动ViewPager时就会产生问题。

于是换成一种NestedScrollView 包含多个RecyclerView, 实现不同的布局

关于截断

android:focusable="true"  
android:focusableInTouchMode="true"

该属性设置可以截断子级控件的默认行为,比如子控件EditText自动获取焦点,关闭键盘, 关闭键盘也可以使用下面这方法:

EditText对象的clearFocus();
InputMethodManager imm = (InputMethodManager)getSystemService(Context.INPUT_METHOD_SERVICE); 
imm.hideSoftInputFromWindow(editMsgView.getWindowToken(), 0);（关闭软键盘。。。）

NestedScrollView嵌套RecyclerView时,RecyclerView 自动指向自己的第一项,此时我们就可以使用

android:focusable="true"  
android:focusableInTouchMode="true"

截断该行为,从NestedScrollView最开始的地方显示。

关于NestedScrollView 嵌套RecyclerView

对于RecyclerView 的adapter中的Item高度, 要设置wrap_content。

滑动问题

1)recyclerview在support library 23.2.0 (or above)以上
2)recyclerview高度设置wrap_content
3)recyclerView.setNestedScrollingEnabled(false)

2016年度个人工作总结报告

发表于 2017-01-02   |     |   阅读次数

时间飞逝，2016年就这样过去了。

这半年多的时间里，我的工作主要是维护易起行android项目,保持项目的活力。

从开始使用GA，进行android、iOS端的数据统计。到后来与营销部的同事讨论起数据的准确性，并不断优化。

从开始的不知cordova为何物，到后来慢慢熟悉，并阅读框架源码。在前端的插件在框架下无法使用时,通过修改源码,使其能够兼容我们的cordova框架,让插件也能够正常使用。

在我们项目随着业务不断扩大,原有旧版本框架已很难继续维护时,能及时的升级我们的cordova框架到最新版本,让版本平缓的过度升级。

在后来丰富我们的项目,使其增加更多的native特性,满足业务需求。

Corodva作为一个跨平台的解决方案，它是我们项目的起始点。它能使版本的迭代更加快速,与backbone框架的结合也使项目的mvc结构更清晰,项目的侵入性也更小。但作为几年前的跨平台解决方案，它存在不少问题，如代码量的巨大,速度慢,不好的使用体验(特别在低端机型)。

关注最近的移动端开发就发现,混合开发将成为常态。其开发周期与版本迭代能力不言而喻。而其中的佼佼者就是react-native,其次是阿里的weex(weex还很不成熟)。

在领导的安排下学习RN, 并尝试开发Demo。其近乎原生的体验,以及相对cordova少不少的代码,体验感觉更好。RN非常适合中小型项目。（但仍有不少坑(目前版本为0.39)）。

很荣幸能够进入广之旅, 并与大家一同共事、成长。

工作就像场马拉松,2016的结束只是路途中的一个拐点。

2017共勉。

okio的作用

okio是作为OkHttp网络请求的io包装库，io文件操作简单。减少了很多io操作的基本代码,并且对内存和cpu使用做了优化。

okio框架结构与源码分析

Okio的简单使用

Okio库的核心是两个接口Sink和Source，这两个接口都继承了Closeable接口；而Sink可以简单的看做OutputStream（将流写入文件），Source可以简单的看做InputStream（从文件读取至流）。而这两个接口都是支持读写超时设置的。其中，Sink声明了write()、flush()、close()、timeout()等方法，Source中声明了read()、close()、timeout(),这些方法包含了对文件的读写及资源的释放。它们各自有一个支持缓冲区的子类接口，BufferedSink和BufferedSource，这两个子接口有一个共同的实现类Buffer,对缓冲区操作。

mac下apktool反编译、重新签名

发表于 2016-08-15   |     |   阅读次数

###增加渠道
当中途运营需要对于某些已经发布的app增加渠道时，多渠道打包就没办法了。本文描述mac下Apk的的修改、重打包、签名。达到增加渠道包的目的。

###反编译
1.打开apktool官网

2.mac系统下：

  1. 点击右键链接另存为apktool
  2. 下载最新的apktool,并更名为apktool.jar。
  3. 把下载下来的apktool和apktool.jar copy到/usr/local/bin目录中。

  4. 按照其他博主的提示，这一步在终端中输入apktool验证apktool环境配置，然而博主在这一步卡住,不知所云：

    $ apktool
zsh: permission denied: apktool
$ sudo apktool
Password:
sudo: apktool: command not found

信号，apktool官网最下面有提示

Note - Wrapper scripts are not needed, but helpful so you don’t have to type java -jar apktool.jar over and over.
  1. cd到我们随便从网上下载的一个Apk目录中，例：demo.apk
  2. 执行反编译命令 apktool d demo.apk
  3. 就可以在当前目录下得到一个demo文件夹。

  4. 于是我们就可以打开其manifest.xml文件，修改

    <meta-data android:name="UMENG_CHANNEL" android:value="channel"/>

###二

  1. 把文件夹重新编译成apk文件

    java -jar /usr/local/bin/apktool.jar b apk
    这时候生成的apk.apk就在 demo/dist 目录中。

  2. 然后我们在利用原来的apk增加签名：

    jarsigner -digestalg SHA1 -sigalg MD5withRSA -tsa -verbose -keystore demo.keystore -signedjar apk.apk apk_release.apk demo.certificate

注意重新签名后的Apk会比原来的apk大一些。原因未知。
